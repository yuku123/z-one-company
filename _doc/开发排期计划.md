---

## 二、排期原则

1. **先基础后业务**：平台基座（gw/ctc/config）→ 能力平台（mq/rpc/ext）→ 业务平台（task/wf/schedule）→ 智能化
2. **先闭环后完善**：先让系统 all-in-one 可跑通，再逐步完善各模块能力
3. **并行可分离**：部分模块可并行开发（不影响彼此）
4. **1人全职估算**：按 5 天/周，8 小时/天估算

## 六、风险点

| 风险 | 影响 | 应对 |
|------|------|------|
| z-wf 从零实现工作流引擎复杂度高 | 延期风险 | 优先用开源（Camunda/Flowable）还是坚持自研？建议先自研摸清原理，后续再迁移 |
| z-ops(Go) 与 Java 体系异构 | 集成风险 | z-ops 只做执行层，通过 HTTP 与 Java 模块通信，不侵入 Java 代码 |
| 单人开发，模块耦合排查慢 | 进度风险 | 每个阶段强制输出接口文档和测试用例，减少 regression |
| 智能化平台（Agent/产品制造器）不确定性高 | scope 蔓延 | 严格按里程碑交付，M4 之前不扩展 scope |

---

## 七、待确认问题

1. **z-wf 自研 vs 开源**：工作流引擎是坚持自研还是引入 Camunda/Flowable？自研工期长但可控，开源工期短但后期迁移成本高。

2. **日志/监控用开源还是自研**：Loki/Prometheus+Grafana 是开源标准件，z-ops 只需拉起；自研工期长。是否优先用开源方案？

3. **M4 Agent 平台 scope**：Agent 平台做到什么程度算「可用」？是只能跑预置 Agent，还是支持用户自定义 Agent？

4. **优先级调整**：是否按此顺序执行，还是有业务压力需要调整优先级（比如先做 z-wf 再补 gw）？
