# 一人公司技术基座 - 从零设计

## 一、愿景

一个人通过这个基座服务自己，方便地产生业务系统，对用户交付。基座具备扩展能力，随业务增长不断扩充。

---

## 二、从零思考：个人公司基座需要什么

### 最高层：协作平台

多个人公司之间如何协作？——**分佣 + 需求匹配**。

| 模块 | 需要做什么 |
|------|-----------|
| 协作平台 | 多个个人公司注册、需求发布、服务商匹配、分佣结算、双向评价 |

---

### 第二层：智能化平台

如何凭空生成一个系统？——**对话式生成 + Agent 制造**。

| 模块 | 需要做什么 |
|------|-----------|
| 产品制造器 | 对话式需求理解 → 系统架构生成 → 代码生成 → 打通 ops 发布 |
| Agent 平台 | 可视化制造 Agent、Skill 管理、MCP 管理、Agent 运行时、LLM 调用、Agent 部署 |

---

### 第三层：应用层

一个人公司交付给客户的业务系统——**任务管理 + 审批 + 沟通 + 内容 + 报表**。

| 模块 | 需要做什么 |
|------|-----------|
| 任务中心 | 任务创建/分配/跟踪/验收 |
| 审批流 | 多级审批、会签/或签、条件路由、审批历史 |
| 日程/提醒 | 日历、倒计时、超时提醒 |
| 消息通知 | 即时消息、站内信、邮件/短信通知 |
| 报表中心 | 数据统计、趋势图、导出 |
| 内容管理 | 文档/知识库、内容发布 |

---

### 第四层：运维平台

系统跑起来后，如何管理它？——**部署 + 监控 + 扩缩容**。
 
| 模块 | 需要做什么 |
|------|-----------|
| 运维平台（z-ops） | Docker 容器拉起/停止/扩容、镜像管理、宿主机监控（CPU/内存/磁盘）、部署流水线 |
| 日志聚合 | 收集各模块日志，统一查询分析 |
| 监控告警 | 指标采集、告警规则、通知 |

---

### 第五层：能力平台

所有上层都会用到的通用能力——**审批流、定时调度、缓存、消息队列、RPC**。

| 模块 | 需要做什么 |
|------|-----------|
| 工作流引擎（z-wf） | 审批流引擎（会签/或签/条件路由），被所有层调用，不只是 task |
| 调度中心（z-schedule） | Cron 定时任务，被所有层调用（z-ctc Token 清理、z-wf 超时提醒、z-task 定时触发） |
| 消息队列（z-mq） | 模块间异步通信，发布订阅 |
| RPC 框架（z-rpc） | 模块间同步调用，可切 HTTP |
| 缓存（z-cache） | JVM 内存缓存，后期兼容 Redis |
| 业务扩展（z-ext） | 平台与业务分离机制，支持对系统打补丁：插件化扩展点、策略注入、业务逻辑热插拔、平台层统一拦截（统一响应/异常/日志）对业务层透明 |

---

### 第六层：平台基座

整个系统运行的前提——**认证、配置、网关、密钥、存储、元数据**。

| 模块 | 需要做什么 |
|------|-----------|
| 4A 认证中心（z-ctc） | 用户管理、角色权限、租户域隔离、JWT Token、OAuth2 |
| 配置中心（z-config） | 配置统一管理、环境隔离、版本管理 |
| API 网关（z-gw） | 路由转发、统一鉴权、限流熔断、注册中心 |
| 密钥管理（z-mist） | 敏感信息加密存储、密钥使用审计 |
| 对象存储（z-oss） | 文件上传下载、后期兼容 S3/MinIO |
| 应用资源元信息（z-meta） | 立项信息、应用关联关系、应用占用资源（CPU/内存/端口）、应用 Key 生命周期的元信息、系统内各项资源的归属和依赖关系 |

---

### 第七层：资源层

被管理的对象——**机器、服务、Key**。

| 模块 | 需要做什么 |
|------|-----------|
| 资源管理 | 物理机/云服务器录入、状态监控、分配调度 |
| 外部服务管理 | LLM API、短信、邮件等外部服务接入 |
| LLM Key 管理 | 调用哪家 LLM、用哪个 Key、成本统计 |

---

### 第八层：基础设施

外部标准组件——**MySQL、Redis、Nginx、Docker**。

| 模块 | 需要做什么 |
|------|-----------|
| MySQL | 持久化存储 |
| Redis | 缓存、Session、分布式锁 |
| Nginx | 反向代理、静态资源 |
| Docker | 容器化部署 |
| DNS / 域名 | 访问入口 |

## 四、与当前设计的对比

### 已有模块（完全匹配）

| 模块 | 对应设计 |
|------|---------|
| z-ctc、z-config、z-gw、z-mist、z-oss、z-meta | 平台基座层 ✓ |
| z-mq、z-rpc、z-ext | 能力平台层 ✓ |
| z-task、z-wf、z-schedule | 应用层/能力平台 ⚠️ 见下方说明 |
| bootstraps/main-starter | All-in-one 打包 ✓ |

### 缺少的模块（需要新建）

| 模块 | 层级 | 需要做什么 |
|------|------|-----------|
| **z-ops** | 运维平台 | Go 微服务，Docker 容器管理、镜像管理、宿主机监控、部署流水线 |
| **日志聚合** | 运维平台 | 收集各模块日志，统一存储查询（可用 Loki/ELK 或自研） |
| **监控告警** | 运维平台 | 指标采集、告警规则、通知（可用 Prometheus/Grafana 或自研） |
| **z-cache** | 能力平台 | JVM 内存缓存，LRU/LFU/TTL，后期兼容 Redis |
| **Agent 平台** | 智能化平台 | 可视化 Agent 制造、Skill/MCP 管理、LLM 运行时 |
| **产品制造器** | 智能化平台 | 对话式生成系统、代码生成、Ops 打通 |
| **协作平台** | 协作平台 | 多租户注册、需求匹配、分佣结算 |
| **日程/提醒** | 应用层 | 日历、倒计时、超时提醒（z-schedule 可部分覆盖） |
| **消息通知** | 应用层 | 即时消息、站内信、邮件/短信（z-mq 可部分覆盖） |
| **报表中心** | 应用层 | 数据统计、趋势图、导出 |
| **内容管理** | 应用层 | 文档/知识库、内容发布 |
| **资源管理** | 资源层 | 物理机/云服务器录入、状态监控、分配调度 |
| **LLM 网关** | 资源层 | 统一管理 LLM 调用、Key 轮换、成本统计（可合并到 z-mist） |

### 需重新归类的模块

| 模块 | 当前设计位置 | 建议位置 | 原因 |
|------|------------|---------|------|
| z-wf | 应用层/运维开发平台 | **能力平台** | z-ctc 等其他层也用审批流，是通用能力 |
| z-schedule | 应用层/运维开发平台 | **能力平台** | z-ctc Token 清理、z-wf 超时提醒等，跨层调用 |
| z-task | 应用层 | **应用层**（不变） | 纯业务平台，给终端用户用 |

## 五、各模块优先级（P0/P1/P2）

### P0（最小基座必须）

| 模块 | 原因 |
|------|------|
| z-ctc | 无认证则无权限，整个系统无法运行 |
| z-gw | 无网关则无统一入口 |
| z-task | 核心业务载体 |
| z-wf | 核心业务载体 |
| z-ops（Go） | 无 ops 则无法管理中间件部署 |
| MySQL | 持久化 |
| z-boot | 共享框架 |

### P1（建议一并建设）

| 模块 | 原因 |
|------|------|
| z-config | 配置统一管理 |
| z-schedule | 定时任务，z-ctc/z-wf 都依赖 |
| z-mist | 密钥管理，LLM 调用必备 |
| z-oss | 文件存储 |
| z-mq | 模块间异步通信 |
| z-rpc | 模块间同步调用 |
| z-cache | 缓存，Redis 不可用时的兜底 |
| z-ops 日志/监控 | 运维必备 |
| z-meta | 元数据管理 |

### P2（逐步建设）

| 模块 | 原因 |
|------|------|
| Agent 平台 | 智能化，依赖 LLM 能力成熟 |
| 产品制造器 | 依赖 Agent 平台 |
| 协作平台 | 依赖业务量 |
| 日程/消息通知 | 可复用通知能力 |
| 报表中心 | 依赖业务数据积累 |
| 内容管理 | 依赖业务场景 |

## 六、z-ops（Go 微服务）详细说明

### 定位

运维平台，用 Go 编写，注册为标准 HTTP 微服务到 z-gw，与 Java 模块地位平等。

### API 路径

```
/api/ops/*  →  z-ops（Go）
/api/ctc/*  →  z-ctc（Java）
/api/task/* →  z-task（Java）
```

### 核心能力

| 能力 | 说明 |
|------|------|
| 容器管理 | docker ps / run / stop / rm，容器生命周期管理 |
| 镜像管理 | docker build / push / pull，镜像构建和分发 |
| Compose 编排 | docker-compose up / down，整组服务启停 |
| 宿主机监控 | CPU / 内存 / 磁盘 / 网络，周期性采集 |
| 中间件自举 | 无外部 MySQL/Redis/Kafka 时，拉起对应 Docker 容器 |
| 部署流水线 | build → test → push → deploy 自动化执行 |
| 日志收集 | 读取容器日志，转发给日志聚合服务 |
| 指标暴露 | 暴露 Prometheus 格式指标，供监控告警使用 |

### 目录结构

```
z-ops/
├── go.mod
├── go.sum
├── main.go                 # 入口，注册 HTTP routes
├── handler/                # HTTP handlers（/api/ops/*）
│   ├── container.go        # 容器管理
│   ├── image.go            # 镜像管理
│   ├── compose.go          # Compose 编排
│   ├── host.go             # 宿主机监控
│   └── deploy.go           # 部署流水线
├── docker/                 # Docker API 封装
├── metrics/                # Prometheus 指标
└── client/                 # 调用其他微服务的 HTTP client
```

## 七、架构疑问

1. **z-ops 的前端控制台**：ops 的运维 UI 是独立项目还是复用现有前端？如果复用，API 路径 `/api/ops/*` 对应的前端页面放在哪个模块下？

2. **日志聚合 + 监控告警**：这两块是用开源组件（Loki/Prometheus）还是自研？如果用开源，ops 负责拉起这些中间件即可；如果自研，需要新建模块。

3. **LLM 网关**：统一管理 LLM 调用、Key 轮换、成本统计——是合并到 z-mist 里，还是独立一个 `z-llm` 模块？

4. **z-wf 和 z-schedule 跨层调用**：这两个模块被所有层调用，放在「能力平台」层，是否需要单独的 starter 供各模块引入？

5. **日程/提醒**：是否直接用 z-schedule 实现，还是需要独立的日程模块？
