# 一人公司技术基座 - 架构整理

## 一、愿景

一个人可以通过这个基座服务自己，方便地产生业务系统并对用户交付。基座具备扩展能力，随业务增长可以不断扩充。

## 四、部署顺序

```
第1步  部署 z-boot（共享 starters）
         ↓
第2步  部署平台基座层（第一要义）
         z-ctc（4A 认证）→ z-config（配置中心）→ z-gw（网关）
         同时部署：z-mist（密钥）、z-oss（存储）、z-meta（元数据）
         ↓
第3步  部署 z-ops（运维平台，Go）
         ↓
第4步  部署 z-task（任务中心）+ z-wf（工作流）
         ↓
第5步  部署 z-schedule（调度）、z-mq（消息队列）
         ↓
第6步  部署 z-rpc、z-ext、z-cache（中间件平台）
         ↓
第7步  Agent 平台 + 产品制造器（智能化）
         ↓
第8步  协作平台（多租户分佣）
```

### 中间件平台与 ops 的依赖关系

**核心逻辑**：ops（运维平台）负责统一管理中间件的部署和生命周期，但中间件平台本身在 ops 尚不存在时也应能独立运行。

```
早期阶段（ops 不存在）：
┌─────────────────────────────────┐
│   中间件自行拉起，独立提供能力    │
│   z-mq 自带消息队列实现          │
│   z-cache 自带缓存实现           │
│   z-rpc 自带 RPC 实现            │
│   无外部依赖，all-in-one 即可运行  │
└─────────────────────────────────┘

ops 出现后（ops 成熟阶段）：
┌─────────────────────────────────┐
│   ops 统一管理中间件生命周期     │
│   z-mq / z-cache / z-rpc 注册到  │
│   ops 的中间件管理平台           │
│   统一运维：扩缩容/监控/告警     │
│   ops 可以拉起标准中间件         │
│   （MySQL/Redis/Kafka）替代自研   │
└─────────────────────────────────┘
```

**演进路径**：

```
Phase 1（自研兜底）：
  无外部中间件 → 中间件模块自带实现 → 系统可运行
       ↓
Phase 2（ops 接管）：
  ops 平台出现 → 接管中间件部署 → 替换为开源标准件
       ↓
Phase 3（中间件平台归位）：
  自研中间件（z-mq/z-rpc/z-cache）→ 兼容开源协议 → 逐步退役
```

**ops 出现前的中间件自启动能力**：

| 自研中间件 | 自启动方式 | 外部兼容目标 |
|-----------|-----------|------------|
| z-mq | 内存消息队列（EventBus） | Kafka / RabbitMQ |
| z-cache | JVM 内存缓存 | Redis |
| z-rpc | 自研 RPC / HTTP 直连 | gRPC / Dubbo |
| z-oss | 本地磁盘存储 | MinIO / S3 |

> 关键原则：**ops 依赖中间件管理，但中间件不依赖 ops 存活**。ops 存在时接管管理，ops 不存在时中间件自举运行。

## 五、最小化基建

一个可以跑起来的最小一人公司平台，至少需要：

| 模块 | 作用 |
|------|------|
| z-boot | 共享框架 |
| z-ctc | 认证+权限（无 CTC 则无租户/域隔离，其他系统无法管理） |
| z-gw | 统一入口 |
| z-task | 任务管理（核心业务载体） |
| z-wf | 审批流（核心业务载体） |
| MySQL | 持久化 |

可选（但建议一并部署）：
- z-config：配置统一管理
- z-schedule：定时任务

## 六、技术方案

### 设计原则

1. **标准中间件优先**：集群内存在标准组件（MySQL、Redis、Nginx）则直接接入，不重复造轮子
2. **自研兜底**：集群内不存在时，对应模块自带实现（z-cache 自研缓存、z-mq 自研消息队列）
3. **向后兼容**：自研模块后期逐步兼容开源标准（z-cache → 兼容 Redis，z-mq → 兼容 Kafka/RabbitMQ）
4. **all-in-one 可用**：无任何外部依赖时，整个系统可单机部署运行

### 基础设施层（外部标准组件）

| 组件 | 用途 | 备注 |
|------|------|------|
| MySQL 8.0 | 持久化存储 | 集群内有则用；无则 z-wf/z-task 等自带 H2 嵌入式 DB 兜底 |
| Redis | 缓存 / Session 共享 | 集群内有则用；无则 z-cache 自研内存缓存兜底 |
| Nginx | 反向代理 / 静态资源 | 集群内有则用；开发环境直接前端 dev server |
| Docker | 容器化 | 部署标准 |

### 各模块技术方案

#### 平台基座层

**z-ctc（4A 认证中心）**
- 技术方案：Spring Security + JWT
- 租户/域隔离：多租户设计，对内一个租户，对外每产品线一个租户，租户下划域（运营域、用户域、VIP 域等）
- 数据库：MySQL（biz_service），多租户通过 tenant_id 字段隔离
- 兜底：无外部 MySQL 时可使用 H2 嵌入式数据库（开发/测试阶段）
- API：/api/ctc/*，Knife4j 文档 /doc.html

**z-config（配置中心）**
- 技术方案：现有实现为基础配置存储，可升级为 Apollo / Spring Cloud Config 模式
- 核心能力：配置的统一管理、版本管理、环境隔离（dev/test/prod）
- 兜底：配置文件本地化管理（application.yml），集群无配置中心时使用
- 数据存储：MySQL（config 表），无 MySQL 时使用本地文件

**z-gw（API 网关）**
- 技术方案：Netty 自研高性能网关（已有），后期可兼容 Spring Cloud Gateway 或 Kong
- 核心能力：路由转发、鉴权、限流、日志、熔断
- 路由规则：按路径前缀路由到后端服务（/api/ctc → z-ctc，/api/task → z-task，/api/wf → z-wf）
- 兜底：开发环境可绕过网关，前端直接请求后端服务端口
- 服务发现：网关内置服务注册表，z-ctc/z-task 等启动时注册，关闭时注销

#### 运维/开发平台层

**z-task（任务中心）**
- 技术方案：任务创建 → 分配 → 执行 → 验收 全生命周期管理
- 任务状态机：创建 → 进行中 → 待验收 → 已完成 / 已取消
- 触发方式：手动创建、定时触发（依赖 z-schedule）、API 触发
- 兜底：无外部 MySQL 时使用 H2 嵌入式数据库

**z-wf（工作流/审批中心）**
- 技术方案：流程定义（bpmn 风格 JSON/YAML）→ 流程实例 → 节点路由 → 审批记录
- 核心能力：会签、或签、条件路由、审批历史、抄送
- 流程引擎：自研（已有），后期可兼容 Camunda / Flowable
- 与 z-task 关系：z-task 的任务可关联 z-wf 的审批流，审批通过后驱动任务状态变更
- 兜底：无 MySQL 时使用 H2

**z-schedule（调度中心）**
- 技术方案：分布式定时任务调度
- 核心能力：Cron 表达式、任务分发、失败重试、任务依赖
- 与其他模块关系：z-task 的定时触发、z-wf 的超时提醒、z-ctc 的定时清理过期 Token
- 兜底：单机内使用 Spring @Scheduled，多节点部署时使用分布式锁（z-rpc 分布式锁或 DB 锁）保证不重复执行

#### 中间件平台层

**z-mq（消息队列）**
- 技术方案：自研消息队列（已有），消息持久化 + 订阅机制
- 消息模型：P2P（点对点）、Pub/Sub（发布订阅）
- 核心能力：消息持久化、消费确认、死信队列、消息重试
- 兼容升级：后期兼容 Kafka / RabbitMQ，接口层抽象，MQ 实现可插拔
- 兜底：同进程内使用内存队列（EventBus），外部 MQ 不可用时保证基本功能

**z-rpc（RPC 框架）**
- 技术方案：自研 RPC（已有），基于 TCP/Netty 长连接
- 核心能力：同步调用、异步调用、单向调用、负载均衡（随机/轮询/一致性哈希）
- 服务注册/发现：z-gw 内置注册中心，服务启动时注册，支持心跳检测
- 序列化：JSON / Protobuf 可选
- 兜底：HTTP via z-gw，当 z-rpc 不可用时降级为 REST 调用

**z-ext（业务扩展）**
- 技术方案：平台与业务分离机制，支持对系统打补丁
- 核心能力：插件化扩展点（SPI）、业务策略注入、逻辑热插拔、统一拦截层（响应/异常/日志）对业务透明
- 与普通工具类的区别：不只是工具类，是一套可让业务层在平台层打补丁的扩展机制

#### 资源管理层

**z-mist（密钥管理）**
- 技术方案：敏感信息（LLM-Key、API-Key、数据库密码等）的加密存储与使用
- 核心能力：密钥的 CRUD、密钥使用审计、密钥轮换（计划中）
- 加密：密钥在 DB 中加密存储，使用时通过 API 获取（不暴露明文）
- 兜底：本地加密配置文件（application-secrets.yml），无 DB 时使用

**z-oss（对象存储）**
- 技术方案：文件上传/下载/管理，支持本地存储和兼容 S3 协议
- 核心能力：分片上传、断点续传、CDN 加速（计划中）
- 存储后端：本地磁盘（已有），后期兼容 MinIO / 阿里云 OSS / AWS S3
- 兜底：使用服务器本地磁盘存储，外部 OSS 不可用时不影响基本功能

**z-meta（应用资源元信息）**
- 技术方案：管理应用立项信息、资源占用（CPU/内存/端口）、应用间关联关系、应用 Key 生命周期、系统内各项资源的归属和依赖
- 核心能力：应用资源录入、关联关系维护、资源变更追踪、依赖分析
- 与 z-ops 关系：z-ops 管理的容器/宿主机资源，z-meta 负责更高层的应用视角抽象
- 兜底：MySQL 存储，无 MySQL 时使用 H2

### 模块间通信协议

```
┌──────────┐     HTTP/REST      ┌─────────┐
│  前端     │ ───────────────►  │  z-gw   │ ────────┐
└──────────┘                    └─────────┘         │
                                                     │ TCP/RPC
                              ┌──────────────────┐  │
                              │  z-ctc / z-task  │◄─┘
                              │  z-wf / z-config │◄─┤
                              └──────────────────┘  │
                                     │             │
                                     │ z-mq (消息) │
                                     ▼             │
                              ┌──────────────────┐  │
                              │  z-schedule       │◄─┘
                              │  z-mist / z-oss  │
                              └──────────────────┘
```

**通信层次**：
- **第一层**：前端 → z-gw（HTTP/REST），所有外部请求统一入口
- **第二层**：z-gw → 后端服务（HTTP via z-gw 路由 或 z-rpc 直连），同进程内直接调用
- **第三层**：模块间异步通知（z-mq），解耦调用
- **降级策略**：z-rpc 不可用 → 降级为 HTTP via z-gw；z-mq 不可用 → 降级为内存 EventBus

### 服务注册与发现

```
服务启动流程：
1. 服务（如 z-task）启动，向 z-gw 注册中心注册（服务名 + IP + 端口）
2. z-gw 维护服务注册表，定时发送心跳检测
3. 心跳超时 → z-gw 自动将该服务从注册表中移除
4. 外部请求经过 z-gw 时，z-gw 按服务名查找注册表进行路由
```

- **注册方式**：服务启动时主动注册，关闭时主动注销
- **心跳机制**：每 30s 发送心跳，3 次无响应视为不可用
- **负载均衡**：z-gw 支持随机、轮询、加权路由策略

## 八、z-gw 核心能力定义

z-gw 是整个系统的统一入口，需要理清它解决的核心问题。

### z-gw 解决的核心问题

| 问题 | 说明 | 解决方案 |
|------|------|----------|
| 服务地址写死 | 每个服务地址写在配置里，迁移/扩容需改配置重启 | 注册中心动态感知 |
| 多实例负载均衡 | 单服务多实例部署时，请求需均匀分发 | 注册中心 + 负载均衡策略 |
| 实例宕机感知 | 某实例挂了，请求不应继续发往该实例 | 心跳检测 + 自动摘除 |
| 统一鉴权入口 | 各服务各自做鉴权，重复且分散 | 网关统一拦截鉴权 |
| 统一出口 | 前端只需知道一个域名，减少耦合 | `/api/{app}/xxx` 统一路由 |

### z-gw 两大模式（可切换）

**模式 A：外部注册中心（近期/过渡）**

```
┌──────────────────────────────────────────────────────┐
│                    Docker Compose                     │
│                                                      │
│    ┌──────────┐        ┌─────────────────────────┐  │
│    │ Consul   │ ◄───── │ z-ctc / z-task / z-wf   │  │
│    │ (开源)   │  注册   │ (服务启动时主动注册)     │  │
│    └────┬─────┘        └─────────────────────────┘  │
│         │                   │                        │
│         │ 服务发现           │ 心跳                   │
│         ▼                   ▼                        │
│    ┌──────────┐        ┌─────────────────────────┐  │
│    │  z-gw    │ ◄───── │ (每 30s 发送心跳)       │  │
│    │          │         │                         │  │
│    └──────────┘         └─────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

- 服务启动 → 注册到 Consul
- z-gw 从 Consul 拉取服务列表，动态路由
- Consul 挂了 → z-gw 读本地缓存（最终降级可用）

**模式 B：z-gw 内置注册中心（未来标准）**

```
┌──────────────────────────────────────────────────────┐
│                    Docker Compose                     │
│                                                      │
│    ┌──────────────────────────────────────────────┐  │
│    │                    z-gw                       │  │
│    │  ┌─────────────┐  ┌──────────────────────┐   │  │
│    │  │  注册中心    │  │      API Gateway    │   │  │
│    │  │ (内置)      │  │   (路由/鉴权/限流)   │   │  │
│    │  └─────────────┘  └──────────────────────┘   │  │
│    └──────────────────────────────────────────────┘  │
│                        ▲                             │
│           ┌────────────┼────────────┐                │
│           │ 注册/心跳   │            │                │
│     ┌─────┴────┐ ┌─────┴────┐ ┌─────┴────┐          │
│     │ z-ctc    │ │ z-task   │ │ z-wf     │          │
│     └──────────┘ └──────────┘ └──────────┘          │
└──────────────────────────────────────────────────────┘
```

- z-gw 内置注册中心，服务直接向 z-gw 注册
- z-gw 集群时，注册数据共享到 MySQL（不丢失）
- 最终形态：z-gw 一体化，Consul/Nacos 退役

### z-gw 注册中心 能力清单

| 能力 | 说明 |
|------|------|
| 服务注册 | 服务启动时上报实例信息（服务名、IP、Port、权重、元数据） |

| 服务发现 | z-gw 按服务名查询可用实例列表 |
| 健康检测 | 心跳机制（TCP 探测 或 HTTP GET /health），3 次无响应视为不健康 |

| 限流 | 基于 IP / 用户 / 服务维度的请求频率限制 |

| 请求日志 | 完整记录请求路径、耗时、响应码 |

### 待建模块详解

#### 1. z-ops（Go 微服务）— 最优先

**定位**：运维平台，用 Go 编写，注册为标准 HTTP 微服务到 z-gw，和 Java 模块地位平等。

**与 Java 项目的关系**：
- z-ops 不依赖任何 Java 代码，独立编译部署
- 通过 z-gw 注册路由：`/api/ops/*` → z-ops
- Java 前端通过 z-gw 调用 ops 的 HTTP API，如同调用 z-ctc 一样
- z-ops 暴露标准的 REST API，内部用 Go 操作 Docker

```
前端 ──HTTP──► z-gw ──路由──► /api/ops/* ──► z-ops（Go）
                               /api/ctc/* ──► z-ctc（Java）
                               /api/task/* ──► z-task（Java）
```

**z-ops 核心能力**：

| 能力 | 说明 |
|------|------|
| Docker 容器管理 | 拉起/停止/重启/扩容容器（MySQL/Redis/Kafka 等） |
| Docker Compose 编排 | 感知 docker-compose.yml，整组启停 |
| 镜像管理 | 镜像构建/推送/拉取 |
| 宿主机监控 | CPU/内存/磁盘/网络监控 |
| 部署流水线 | 执行 build → push → deploy 流水线 |
| 中间件自举 | 无外部中间件时，ops 可拉起标准 MySQL/Redis/Kafka 容器 |
| ops 控制台 | 运维人员 Web UI（可选，Go 写轻量页面 或 复用前端） |

**z-ops 目录结构（建议）**：

```
z-ops/                      # Go 微服务，顶级目录
├── go.mod
├── go.sum
├── main.go               # 入口，注册 HTTP routes
├── handler/              # HTTP handlers（/api/ops/*）
├── docker/               # Docker API 操作层
├── compose/              # Docker Compose 编排
├── image/                # 镜像管理
├── host/                 # 宿主机监控
└── client/               # 调用其他微服务的 HTTP client（z-gw/z-ctc）
```
