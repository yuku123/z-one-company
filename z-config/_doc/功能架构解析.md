# z-config 功能架构解析

## 1. 项目定位

**z-config** 是一个对标 Nacos 的分布式配置管理和服务发现平台。提供以下核心能力：

- **配置管理**：动态配置推送、版本管理、灰度发布
- **服务发现**：服务注册、健康检查、负载均衡
- **动态DNS**：服务订阅、实例推送

## 3. 核心功能模块

### 3.1 配置管理模块

| 功能点 | 实现类 | 关键方法 | 说明 |
|--------|--------|----------|------|
| 配置保存 | `ConfigServiceImpl` | `saveConfig()` | 支持插入/更新，三主键(namespace+group+dataId)唯一 |
| 配置获取 | `ConfigServiceImpl` | `getConfig()` | 从数据库查询返回配置内容 |
| 配置推送 | `ServerBusinessHandler` | `notifyAllClients()` | 配置变更后推送到所有监听客户端 |
| 分页查询 | `ConfigServiceImpl` | `pageConfig()` | 支持分页的配置列表查询 |

**配置变更推送流程：**
```
1. 管理员调用 POST /config/saveConfig
2. ConfigServiceImpl.saveConfig() 保存到数据库
3. 调用 ServerBusinessHandler.notifyAllClients()
4. 遍历所有连接的Netty Channel
5. 向监听了该配置的客户端推送变更通知
6. 客户端收到通知后重新拉取配置
```

### 3.2 服务发现模块

| 功能点 | 实现类 | 关键方法 | 说明 |
|--------|--------|----------|------|
| 服务注册 | `ZNamingServiceImpl` | `registerInstance()` | 支持分组、集群、命名空间 |
| 服务注销 | `ZNamingServiceImpl` | `deregisterInstance()` | 级联删除实例、更新cluster_map |
| 服务查询 | `ZNamingServiceImpl` | `getAllInstances()` | 查询所有/健康实例 |
| 服务订阅 | `ZNamingServiceImpl` | `subscribeService()` | 消费实例订阅服务变更 |
| 负载均衡 | `ZNamingServiceImpl` | `selectOneHealthyInstance()` | 随机选择健康实例 |

**服务注册数据模型：**
```
服务名格式: {group}@@{serviceName}
实例ID格式: {serviceId}@@{ip}:{port}

z_service_info: 服务元数据
  - service_name: "DEFAULT_GROUP@@order-service"
  - namespace: ""
  - cluster_map: {"DEFAULT": ["192.168.1.10:8080", "192.168.1.11:8080"]}

z_instance: 实例列表
  - instance_id: "1@@192.168.1.10:8080"
  - service_id: 1
  - ip: "192.168.1.10"
  - port: 8080
  - healthy: 1
  - cluster_name: "DEFAULT"
```

### 3.3 长连接通信模块

| 组件 | 类 | 职责 |
|------|-----|------|
| Netty服务端 | `NettyServerComponent` | 监听12888端口，管理客户端长连接 |
| 协议编码器 | `CustomProtocolEncoder` | 自定义协议编码（解决粘包） |
| 协议解码器 | `CustomProtocolDecoder` | 自定义协议解码（长度字段校验） |
| 心跳处理器 | `HeartbeatHandler` | 维护连接活性检测 |
| 业务处理器 | `ServerBusinessHandler` | 处理配置监听、变更推送 |

**自定义通信协议：**
```
协议格式（基于Netty LengthFieldBasedFrameDecoder）:
+--------+----------------+---------+
| 长度域  |    类型域       |  数据域  |
|4 bytes |    4 bytes     | N bytes |
+--------+----------------+---------+

类型域定义（CommandType）:
- 0x01: 心跳请求 (HEARTBEAT_REQUEST)
- 0x02: 心跳响应 (HEARTBEAT_RESPONSE)
- 0x03: 配置拉取请求 (CONFIG_POLL)
- 0x04: 配置推送通知 (CONFIG_PUSH)
- 0x05: 配置变更响应 (CONFIG_CHANGED)
```

**长连接工作原理：**
```
1. 客户端启动时通过Netty连接服务端:12888
2. 发送心跳包维持连接活性（默认5秒间隔）
3. 对关注的配置发送CONFIG_POLL请求，带上当前MD5
4. 服务端比对MD5：
   - 一致：保持连接等待（长轮询）
   - 不一致：立即返回新配置
5. 配置被修改时，服务端主动推送CONFIG_PUSH到所有监听客户端
6. 客户端收到推送后重新拉取配置并触发回调
```

## 5. 数据库设计

### 核心表结构

```sql
-- 配置管理核心表
z_config_info         -- 主配置表（存储实际配置内容）
z_config_info_beta    -- Beta环境配置（灰度发布）
z_config_info_history -- 配置历史版本（变更记录）

-- 服务发现核心表
z_service_info        -- 服务元数据（服务名、集群映射）
z_instance            -- 服务实例（IP、端口、健康状态）
z_cluster             -- 集群定义（健康检查配置）
z_subscription        -- 服务订阅关系（消费者-服务关联）

-- 权限控制表
z_users               -- 用户表
z_roles               -- 角色表
z_permissions         -- 权限表
```

## 6. 关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 长连接协议 | Netty自定义协议 | 高并发、低延迟、支持心跳检测 |
| 数据存储 | MySQL + MyBatis-Plus | 成熟稳定、事务支持、易于维护 |
| 服务模型 | 分组+集群+命名空间 | 灵活的多租户隔离、环境管理 |
| 配置推送 | 长轮询+主动推送 | 实时性好、服务端控制推送时机 |
| 客户端集成 | Spring Boot Starter | 零配置、注解驱动、易于使用 |
